# Комментарии — кратко и по существу

> Claude-эквивалент [`.cursor/rules/comments-style.mdc`](../../.cursor/rules/comments-style.mdc). Применяется при работе с `**/*.ts`, `**/*.cjs`, `**/*.js`, `**/*.md`, `**/*.mdc`.

Объясняй неочевидное (зачем, ограничения, компромиссы), не пересказывай код — см. «Default to writing no comments» в [CLAUDE.md](../../CLAUDE.md) / корневом системном промпте.

## Код (`src/**/*.ts`, `tests/**/*.cjs`, `*.cjs`)

- Inline-комментарий (после кода на той же строке) — **один пробел** до `//`, затем `// ` и текст; не выравнивать колонку `//` между соседними строками (выравнивание — только в fenced-командных блоках документации, см. ниже).
- Комментарий перед кодом (на отдельной строке) — на уровне отступа следующей строки кода, как обычная строка.
- Комментарии — на русском, лаконично; без пересказа «что делает код» (для этого достаточно имён — [ts-style.md](ts-style.md) / [js-style.md](js-style.md)).
- Не редактировать tsc-артефакты (`rules/*.js`, `domain/*.js`, корневые `.js`) — комментарии правь только в `src/**/*.ts`.

## Документация (все `*.md` / `*.mdc` проекта)

Inline-комментарии в **командных** fenced-код-блоках (`bash`, `sh`, `shell`,
`powershell`, `pwsh`, `bat`, `cmd`) держим на **одном уровне** (выровнены в
одну колонку) по правилу:

- выравнивание считается по локальной группе строк внутри блока (группы разделяются пустой строкой);
- опорная длина — длина **самой длинной строки группы без комментария** (для строк с `# ...` берётся текст до пробелов перед `#`; для строк без `#` — вся строка); строки, целиком состоящие из комментария (начинаются с `#` после отступов), в опорную длину не входят и не проверяются;
- колонка `#` = опорная длина + 4 пробела;
- если в группе меньше двух строк с inline-комментарием — выравнивание не требуется.

Правило действует для всех markdown-документов репозитория (`README.md`,
`AGENTS.md`, `CLAUDE.md`, `.cursor/rules/*.mdc`, `.claude/rules/*.md`,
`markdownlint-examples/*` и т. д.), кроме индентированных (не fenced) блоков
и блоков на языках вне списка выше — там нет автоматической проверки.

Проверка этого правила автоматизирована тестом [`tests/test-markdown-comments.cjs`](../../tests/test-markdown-comments.cjs) и входит в `npm test`.

Пример (fenced `bash`, опорная длина 28, колонка `#` = 32):

```bash
npm test                        # запуск тестов
npm run lint:md -- README.md    # точечный lint
```

Не выравнивай по случайно завышенному отступу; опорная колонка считается как
`длина_самой_длинной_строки_группы + 4`, не по вкусу.

## Таблицы (Markdown)

Столбцы GFM-таблиц во всех `*.md` / `*.mdc` проекта выровнены по ширине по правилу:

- ширина столбца — длина **самой длинной** ячейки этого столбца среди всех строк таблицы (заголовок и тело, разделительная строка не считается), но не меньше 3;
- содержимое ячейки дополняется пробелами справа до этой ширины; вокруг дополненного содержимого — по одному пробелу перед `|` и после (`| текст |`);
- разделительная строка — дефисы без пробелов, длиной `ширина + 2` (та же длина, что и у заполненной ячейки данных вместе с её пробелами-паддингами), без выравнивающих `:`;
- каждая таблица выравнивается независимо; строки таблицы должны начинаться и заканчиваться `|` (без него — не таблица для этого правила);
- экранированный `\|` внутри ячейки не считается разделителем столбцов.

Проверка этого правила автоматизирована тестом [`tests/test-markdown-tables.cjs`](../../tests/test-markdown-tables.cjs) и входит в `npm test`.

## Чего не делать

- Оставлять «висячие» построчные `//`/`#`-комментарии, пересказывающие соседний код;
- Смешивать выровненную и невыровненную колонку `#` внутри одной группы командного fenced-блока;
- Комментировать генераторы/шаблоны (`scripts/sync-cli2-config.cjs`) без пометки, что это текст **вывода**, а не комментарий к самому generator-коду;
- Оставлять невыровненные столбцы GFM-таблиц после ручной правки строки (добавить/убрать ячейку) — перевыравнивать всю таблицу.
