# Формат README (`## Обзор`)

> Claude-эквивалент [`.cursor/rules/readme-format.mdc`](../../.cursor/rules/readme-format.mdc). Применяется при работе с `README.md`.

Вводная часть [`README.md`](../../README.md) приведена к единому внутреннему
формату. Правя вводную часть — сохраняй эту форму, а не произвольный текст.

## Скелет вводной части

1. `# markdownlint-custom` — H1, без текста между заголовком и `## Обзор`
   (никакого «голого» вступительного абзаца перед секцией).
2. `## Обзор` — первая секция документа, ровно два блока внутри:
   - **Абзац-описание**: одно-два предложения вида «Проект —
     **<жирный обобщающий ярлык>** …», объясняющих архитектурную суть
     проекта на верхнем уровне (единый конфиг для IDE и CLI, путь от
     TypeScript-класса правила до работающей проверки), БЕЗ перечисления
     конкретных custom-правил — это задача таблицы дальше по документу.
   - **Список «как использовать»**: фраза `Репозиторий можно использовать
     как:` и буллет-лист из трёх пунктов формата `**<Роль>** — <описание>`.
     Канонические роли (порядок и точная формулировка могут отличаться по
     смыслу проекта, но набор фиксирован — все три пункта обязательны):
     1. **Референс архитектуры** — как оформить custom-правило markdownlint
        (`BaseRule`, `parser`, `onError`), не изобретая контракт с нуля;
     2. **Готовый набор правил** — custom-правила можно подключить как
        есть;
     3. **Отправную точку** — скопировать структуру (`src/`,
        `markdownlint-examples/`, `tests/`, `.markdownlint-cli2.jsonc`) под
        свои правила и адаптировать/отключить готовые.
3. После списка ролей — конкретика верхнего уровня (единый конфиг
   `.markdownlint-cli2.jsonc`, расположение исходников/runtime, entry
   points `markdownlint-rules.js`/`markdownlint-hlprs.js`), но это уже не
   часть формулы «Обзора», а её продолжение.

## Дальше по документу (общий каркас)

После `## Обзор` README следует единому каркасу секций:

- `## Требования` — версии/зависимости, буллет-лист.
- Секции по доменным единицам проекта (в этом репозитории — `## Переносы
  строк (LF)` про кроссплатформенную специфику, `## Быстрый старт` про
  минимальные команды `npm install`/`npm test`, `## Локальная проверка без
  IDE` про bin-скрипты, `--help`, passthrough) — состав, имена и порядок по
  факту проекта, не унифицируются.
- Секция про custom-правила (`## Правила` или аналог) — таблица правил,
  специфичная для проекта.
- Опционально — секция перекрёстных ссылок на связанную документацию в
  конце.

## Что НЕ унифицируется

- Конкретные названия секций после «Обзора» (кроме самой формулы) — они
  отражают специфику проекта, а не общий канон.
- Таблицы форматируются по [`comments-style.md`](comments-style.md)
  (выравнивание столбцов GFM-таблиц) — эта правка не про формат таблиц, а
  про форму вводного текста.
- Полные политики custom-правил не копируются в README — см.
  [`docs-consistency.md`](docs-consistency.md) («Роли артефактов»): в
  README — кратко, канон — в
  [`markdownlint-project.md`](markdownlint-project.md).

## Проверка перед завершением правки README

- [ ] `## Обзор` идёт сразу после `# markdownlint-custom`, без текста между
      ними.
- [ ] Абзац-описание — не перечисление custom-правил, а архитектурная суть.
- [ ] Список «как использовать» — ровно три пункта с жирными ролями
      (референс/готовый набор/отправная точка).
- [ ] Остальные секции README не потеряли контент при правке вводной
      части.
