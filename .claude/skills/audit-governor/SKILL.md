# Audit Governor

## Режимы запуска

Используй только один из двух режимов:

- `audit changed` — аудит внесённых правок.
- `audit full` — полный аудит проекта.

Если режим не указан, по умолчанию используй `audit changed`.

## Обязательные источники истины

Перед аудитом прочитай:

- `AGENTS.md`
- `README.md`
- `package.json`
- `.markdownlint-cli2.jsonc`
- `CLAUDE.md` (корневой) и все `.claude/rules/*.md`
- `checklist.md` этого skill

## Область проверки

### Для `audit changed`

Проверяй:

- unstaged и staged изменения;
- untracked файлы;
- все коммиты текущей ветки относительно реальной base-ветки (`main` или
  `master`);
- все связанные изменения в коде, тестах, примерах, документации, правилах,
  агентных инструкциях.
- не ограничивайся только последним коммитом.

### Для `audit full`

Проверяй весь репозиторий целиком по тем же критериям качества и
консистентности.

## Что обязательно валидировать

- абсолютный coverage всех пластов: код (`src/`, tsc-артефакты в корне),
  тесты (`tests/`), примеры (`markdownlint-examples/`), документация
  (`README.md`, `AGENTS.md`, `CLAUDE.md`), комментарии, правила проекта
  (`.claude/rules/*.md`, `.cursor/rules/*.mdc`);
- высокий инженерный уровень реализации: корректная декомпозиция, SRP, DRY,
  минимализм, простота и читаемость (`ts-dev.md`/`js-dev.md`);
- каталог custom `names` идентичен в коде (`markdownlint-rules.ts` /
  `markdownlint-rules.js`), `.markdownlint-cli2.jsonc`, `AGENTS.md`,
  `README.md` и обоих каталогах правил;
- контракт `onError({ lineNumber, detail, context? })`: `lineNumber` с 1,
  `detail` из `src/details.ts`, `context` — полная строка для
  indent-правил или `trim` для prose-правил (`ts-style.md`);
- `parser: "none" | "micromark"` каждого правила соответствует политике в
  `markdownlint-project.md`;
- контракт `.markdownlint-ignore`: якорность многосегментных паттернов от
  корня репозитория, неякорность однос сегментных (`platform-scripts.md`);
- контракт веб-хука `notify.js`: `MDLINT_WEBHOOK_URL` обязан быть
  `https://`, короткое сообщение об ошибке (не список нарушений), таймаут
  `2000` мс, fire-and-forget (не влияет на код возврата CLI);
- комментарии: минимальность, актуальность, корректность, выравнивание
  inline-`#` в командных fenced-блоках docs и GFM-таблиц по
  `comments-style.md`;
- отсутствие необоснованных suppression-комментариев
  (`// eslint-disable`, `@ts-ignore`, `@ts-expect-error`) без задокументированного
  обоснования, аналогичного существующим в `src/domain/micromark-parse.ts` /
  `micromark-token-utils.ts`;
- симметрия и идентичный порядок трёх листингов правил (`CLAUDE.md`,
  таблица «Правила репозитория» в `AGENTS.md`, карта в `rules-sync.md`) —
  проверяется тестом `tests/test-rules-consistency.cjs`.

Если для прохождения проверок нужны изменения в `CLAUDE.md`, `.claude/rules/*.md`
или `AGENTS.md`, вноси минимальные согласованные правки и повторяй цикл проверок.

## Обязательный цикл проверок

Используй команды из `checklist.md` и добивайся полного green:

- `npm test`
- `npm run check`
- `npm run lint:md -- README.md AGENTS.md`

После каждой серии исправлений запускай проверки повторно до полного green.
После собственных исправлений выполняй повторный аудит тем же режимом
(`audit changed` или `audit full`) до полного green.

Отдельно контролируй, что в `npm test` проходит
`tests/test-rules-consistency.cjs` (набор и порядок правил синхронны между
`CLAUDE.md`, `AGENTS.md` и `rules-sync.md`, у каждого файла-правила — парный
`.mdc`/`.md`, blockquote-ссылка и frontmatter; точные инварианты — в
`.claude/rules/rules-sync.md`, не дублируй их здесь), а также
`tests/test-markdown-tables.cjs` (выравнивание GFM-таблиц по всему
проекту) — точные параметры выравнивания в `.claude/rules/comments-style.md`.

## Политика исправлений

- Исправляй автоматически всё, что можно исправить корректно и безопасно.
- Не вноси косметические изменения без пользы.
- Не ломай публичные контракты без крайней необходимости.
- Не ослабляй кроссплатформенные гарантии bin-скриптов.
- Не оставляй частичную проверку: завершай аудит только после валидации
  всех пластов и повторного цикла проверок.
- Если выявлено, что правила проекта или `AGENTS.md` недостаточны для
  прохождения проверок, обновляй их минимально и согласованно с фактическим
  поведением кода.

## Формат финального ответа

Возвращай кратко:

1. список изменений по файлам;
2. результаты обязательных проверок;
3. что исправлено для консистентности;
4. подтверждение DoD.
